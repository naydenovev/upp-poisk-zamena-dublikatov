## Context

Обработка «Поиск и замена дублирующихся элементов» сохранена как внешняя (`ПоискИЗаменаДублирующихсяЭлементов.epf`) из типовой конфигурации УПП Украина 1.3 (BAF 8.3.23.2299, целевая платформа заказчика 8.3.10.2772). Все формы обработки — **обычные** (не управляемые), поэтому их модули хранятся в бинарном `Form.bin` и правятся только после распаковки через `v8unpack` (`python -m v8unpack -E`), а не через выгрузку конфигуратора (`/DumpExternalDataProcessorOrReportToFiles` оставляет обычные формы бинарными).

Вся логика находится в модуле формы `Форма` (`Form/Форма/Form.obj.bsl`, ~1970 строк). Ключевые существующие элементы:

- `мРезультатыПоиска` (реквизит формы, Экспорт) — таблица групп дублей: `НомерГруппы`, `Правильный` (Булево), `Ссылка`.
- `НайденныеОбъекты` (реквизит формы, табличная часть/значение) — текущая отображаемая группа: `Ссылка`, `Правильный`, реквизиты для сравнения.
- `ВыполнитьЗаменуЭлементов(ПравильныйЭлемент, ТаблицаСсылок, ...)` — вручную проходит по документам/справочникам/регистрам/последовательностям и заменяет ссылку на «правильный» элемент (замена не через какой-либо штатный `ЗаменитьСсылки` — такого метода в этой версии платформы нет).
- `КоманднаяПанельНайденныеСсылкиУдалитьЭлементы` (кнопка «Удалить элементы», ручной путь, вкладка «Найденные ссылки») — **точка изменения**. Сейчас: доступна только когда предварительный платформенный `НайтиПоСсылкам` вернул 0 ссылок; после этого — прямой `Объект.Удалить()` в `Попытка/Исключение` без пометки, без структурированного отчёта.
- `АвтозаменаЭлементов(НепосредственноУдалятьОбъекты)` (пакетный путь, кнопка «Автозамена») — не меняется в рамках этого change.

В той же базе (типовая конфигурация УПП, `DataProcessors.УдалениеПомеченныхОбъектов`) есть готовый `ManagerModule` с экспортной функцией:

```
Функция УдалитьПомеченныеОбъекты(знач Удаляемые, ТипыУдаленныхОбъектовМассив) Экспорт
```

— принимает явный массив ссылок (не сканирует всю базу), итеративно вызывает платформенный `УдалитьОбъекты(Массив, Истина, НайденныеДанные)`, устанавливает монопольный режим, учитывает `ОбщегоНазначения.ПолучитьОбщийСписокИсключенийПоискаСсылок`, возвращает структуру `Статус/Значение{Найденные, НеУдаленные}`.

## Goals / Non-Goals

**Goals:**
- Заменить прямой `Объект.Удалить()` в ручном пути («Удалить элементы») на: пометка на удаление → контролируемое удаление через типовой `Обработки.УдалениеПомеченныхОбъектов.УдалитьПомеченныеОбъекты`.
- Ограничить кандидатов на удаление строго элементами текущей группы поиска/замены (`НайденныеОбъекты`, `Правильный = Ложь`) — без обращения к глобальному списку помеченных объектов базы.
- Показать пользователю итоговый отчёт (помечено / удалено / не удалено + причина).
- Не откатывать пометку на удаление для объектов, которые не удалось удалить — они остаются помеченными для последующей штатной «Удаление помеченных объектов».

**Non-Goals:**
- Не меняется пакетный путь «Автозамена» (`АвтозаменаЭлементов`) — остаётся с текущим прямым удалением.
- Не меняется сам алгоритм поиска дублей и замены ссылок (`ВыполнитьЗаменуЭлементов`, `НайтиПоРавенствуРеквизитов`, `НайтиПоСовпадающимСловам`).
- Не создаётся собственная реализация контроля ссылочной целостности — используется типовая.
- Не переносится доработка в основную конфигурацию — только во внешнюю обработку.

## Decisions

**1. Переиспользовать `Обработки.УдалениеПомеченныхОбъектов.УдалитьПомеченныеОбъекты`, а не писать свой контроль.**
Альтернатива — вызывать платформенные `ПроверитьВозможностьУдаления`/`УдалитьОбъекты` напрямую, не завязываясь на наличие конкретной обработки в целевой базе. Отклонено: типовая обработка есть практически в любой конфигурации 1С (включая УПП), а её менеджер-модуль уже реализует итеративный алгоритм и учёт исключений поиска ссылок — переизобретать его означало бы дублировать типовую логику с риском разойтись с поведением штатного инструмента, который пользователь и так знает.

**2. Кандидаты на удаление — только текущая группа `НайденныеОбъекты` (`Правильный = Ложь`), а не `Обработки.УдалениеПомеченныхОбъектов.ПолучитьПомеченныеНаУдаление()`.**
Явное требование: обработка не должна трогать объекты, помеченные на удаление кем-то другим или в другое время. `УдалитьПомеченныеОбъекты` принимает явный массив ссылок параметром — это позволяет вызвать типовой контроль без обращения к глобальному сканированию базы.

**3. Изменение только в ручном пути (кнопка «Удалить элементы»).**
Пакетный путь «Автозамена» уже имеет собственную (более простую) пред-проверку через `НайтиПоСсылкам` сразу после массовой замены по всем группам разом — перевод его на пометку+типовой контроль потребовал бы отдельного проектирования (транзакционность на несколько групп, поведение при частичном успехе автозамены) и не запрошен пользователем в этой итерации.

**4. Не удалённые объекты остаются помеченными на удаление.**
Это ровно то же самое состояние, в котором окажется объект, если пользователь пометит его вручную и запустит штатную «Удаление помеченных объектов» — предсказуемо для пользователя 1С и не требует отдельного «отката пометки».

**5. Инструментарий разработки: `v8unpack` вместо конфигуратора для правки BSL.**
Обычные формы этой обработки сериализуются в `Form.bin` целиком (включая модуль). Выгрузка конфигуратором (`/DumpExternalDataProcessorOrReportToFiles`) не декомпилирует обычные формы в текст. `v8unpack -E` — декомпилирует (`Form/Форма/Form.obj.bsl`), `v8unpack -B` — собирает обратно в EPF. Правка ведётся только в `.bsl`.

## Risks / Trade-offs

- **[Риск]** Целевая база не содержит `DataProcessors.УдалениеПомеченныхОбъектов` (нестандартная/урезанная конфигурация) → обращение к нему упадёт с ошибкой времени выполнения.
  **Митигация:** на старте процедуры — мягкая проверка наличия объекта метаданных (`Метаданные.Обработки.Найти("УдалениеПомеченныхОбъектов")`); при отсутствии — понятное сообщение пользователю без падения формы.
- **[Риск]** У пользователя нет права `ИнтерактивноеУдалениеПомеченных` на конкретный справочник → `УдалитьПомеченныеОбъекты` либо тихо пропустит объект (по построению типового менеджер-модуля), либо потребует привилегированного режима.
  **Митигация:** отчёт явно покажет объект как «не удалён»; отдельная проверка прав до пометки не требуется — типовой механизм уже это учитывает через `ПравоДоступа` в `ПолучитьПомеченныеНаУдаление` (для нашего сценария не используется, но сам `УдалитьОбъекты` уважает права).
- **[Trade-off]** Пометка на удаление — это запись объекта (`Объект.Записать()`) для каждого дубля перед попыткой удаления. Если дальнейшее удаление не удастся, в базе останется физически изменённое состояние (`ПометкаУдаления = Истина`) — это осознанный и подтверждённый пользователем результат (см. Decisions §4), а не побочный эффект.
- **[Риск]** Сборка через `v8unpack` может не пройти round-trip идентично (см. `upp-ordinary-forms-epf-techniques` — риски правки обычных форм). **Митигация:** после сборки — контрольная распаковка нового EPF и сверка BSL-диффом; финальная приёмка — пользователем в реальном интерфейсе (открыть форму, проверить кнопки) на демо-базе.

## Migration Plan

1. Правка `src_epf/Form/Форма/Form.obj.bsl`: новая процедура пометки+контроля+удаления, замена вызова в `КоманднаяПанельНайденныеСсылкиУдалитьЭлементы`.
2. Сборка: `python -m v8unpack -B src_epf ПоискИЗаменаДублирующихсяЭлементов.epf` (в НОВЫЙ файл, не поверх оригинала, до подтверждения).
3. Смоук-тест: контрольная распаковка нового EPF, сверка структуры; открытие в толстом клиенте на демо-базе (`1cv8.exe ENTERPRISE /F ... /Execute`).
4. Функциональная проверка на демо-базе: найти реальные дубли в тестовом справочнике, выполнить замену, нажать «Удалить элементы», проверить отчёт и фактическое состояние объектов (удалены / помечены).
5. Приёмка пользователем на демо-базе.
6. Публикация в отдельный репозиторий на GitHub (`naydenovev`) с README (с пометкой о разработке при помощи Claude Code).

Откат: правка затрагивает только внешний EPF-файл, конфигурация базы не меняется — откат = не использовать новый EPF / вернуться к ранее сохранённому оригиналу `ПоискИЗаменаДублирующихсяЭлементов.epf`.
